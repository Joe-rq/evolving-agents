代码生成进化：从错误代码到全测试通过 生产场景 · Roadmap→已验证

同一套引擎，第六个领域。代码生成是 LLM 时代最常见的 Agent 任务——这里展示引擎怎么从"接近但错"的代码嫁接修复。配 experiments/codegen_evolution.py

两条进化链：嫁接一个修复，从部分通过到全通过

问题：返回列表元素的平方和。测试用例 [1,2,3]→14, [0]→0, [1,1,1]→3, [2,3]→13
链一：用错运算符（×2 而非 x²）
0.25
return sum(x*2 for x in lst)
4 个测试只过 1 个 — 用了 ×2 而不是 x²，[1,2,3] 算出 12 而非 14
↓ graft: x*2 → x**2 — 把乘法改成幂运算
1.00
return sum(x**2 for x in lst)
✅ 4 个测试全过 — 正确的平方和
链二：忘记平方（直接求和）
0.50
return sum(x for x in lst)
4 个测试只过 2 个 — 求和而非平方和
↓ graft: +**2 — 在求和表达式里加上平方
1.00
return sum(x**2 for x in lst)
✅ 4 个测试全过
同一个 bug，两条路径，同一个正确答案。 引擎从"接近但错"的代码出发，嫁接一个修复（改运算符 / 加缺失的平方），跳到全测试通过。这正是 LLM 代码 Agent 最需要的能力——不重写，而是在接近正确的版本上修。

和 LLM 的配合

引擎不替代 LLM——它帮 LLM 避免重复犯错。
阶段 没有痕进 有痕进
LLM 生成5 个方案，含已知会错的5 个方案，已知死路被屏蔽
执行测试全跑一遍（浪费）只跑没试过的
记录失败对话窗口里，下次忘写进账本，下次自动跳过
下一轮LLM 可能又犯同一个错引擎提醒"这类 API 上次崩了"

诚实边界

✅ 证明了 ⚠️ 局限
引擎在第六个领域（代码生成）跑通；进化链清晰：从部分通过嫁接修复到全通过。LLM 场景可接入。 任务是编程题级别（LeetCode easy），非真实生产代码库。graft 是预定义的修复模式，非 LLM 自主修复。
定位：代码生成证明了引擎在LLM 时代最常见场景上的可接入性。从 Roadmap 变成了已验证——但任务规模还是玩具级，生产级集成需要对接真实编译/测试流水线。
