## 五、项目总览回答：为什么做、怎么做、怎么落地

> **使用场景**：评委问"你这个项目到底为什么做？价值在哪里？怎么落地？"时，用这一节。它不是逐页口播稿，是总览回答稿。

### 1. 为什么做

现在很多 Agent 看起来会试错，但其实不太长记性。

常见的问题是：这次失败修掉了，下一次换个壳又回来。历史要么被丢掉，要么被塞进 prompt 里碰运气。失败没有进入决策系统，所以每一轮都像从零开始。

痕进要解决的不是"让 Agent 第一次就做对"，而是：

> 让 Agent 第一次做错之后，第二次少走同一条错路。

这就是项目的出发点：**别把失败当垃圾，把它变成资产。**

### 2. 做什么

痕进不是一个新的 Agent 平台，也不是一个更大的 prompt 模板。

它是一个嵌入到搜索循环里的**搜索记忆引擎**：

> 记录试过什么，识别失败模式，把失败投影成规则，影响下一轮搜索。

一句话：

> 痕进让 Agent 的试错过程可记忆、可审计、可复用。

它关心的不是“最后答案是什么”，而是“这轮为什么失败、下轮应该避开什么、什么时候该换方向”。

### 3. 如何做

它把 Agent 的试错过程拆成一个通用循环：

1. **候选生成**：宿主 Agent 或业务系统生成一批候选方案。
2. **候选分类**：adapter 判断候选属于哪个策略族，也就是 `family`。
3. **失败记录**：每次尝试都写入 `Memory`：成功/失败、特征、失败模式、证据标签。
4. **模式学习**：系统不只记"这个候选失败"，而是归纳"这一类候选容易失败"。
5. **规则投影**：从失败账本里生成可审计规则，比如 block / penalize，并带 `evidence_ids`。
6. **下一轮搜索**：引擎用这些规则重新排序、屏蔽死路、奖励新颖方向。
7. **结构 pivot**：如果某类策略反复失败，引擎建议换搜索空间，而不是继续调参数。

技术上，`core/` 是领域无关引擎；每个领域只需要写 adapter，告诉引擎：

- 怎么分类候选
- 怎么抽特征
- 怎么识别失败 motif
- 怎么判断成功/失败
- 怎么执行或评估结果

但这里要说实话：**难点不在 core，而在 adapter。** 如果一个领域的失败模式定义得很烂，引擎也学不到什么。这点不能装傻，评委一定会问。

### 4. 如何落地

它适合这类任务：

- Agent 会反复试错
- 失败能被客观定义
- 同类失败会重复出现
- 候选方案能被结构化描述
- 每次尝试有可记录的结果

比如：

- 代码生成：测试失败、异常类型、API 使用错误
- SQL 优化：慢查询、笛卡尔积、缺索引、过宽查询
- RAG 检索：召回失败、chunk 太大、阈值错误、rerank 缺失
- 量化因子挖掘：因子失败、拥挤、相关性过高
- 规则合成：误召回、漏召回、规则过宽
- 自动化实验：参数组合失败、结构方向失败

落地方式不是替换原系统，而是插入原有循环：

> 生成候选 → 痕进排序/过滤 → 执行评估 → 痕进记录失败 → 下一轮更新搜索。

所以它更像一个“记忆和决策中间层”，不是一个完整应用。

### 5. 可行性在哪里

可行性来自四点：

1. **机制已经实现。** Memory、Rules、MetaLearner、Evolver、Novelty 都已经在代码里，不是 PPT 概念。
2. **核心引擎领域无关。** 不同场景只换 adapter，`core` 不需要为每个业务重写。
3. **它不依赖 LLM。** 引擎本身是本地、确定性、零 API 成本的结构化数据操作。LLM 可以负责生成候选，但痕进负责"筛和记"。
4. **有公开边界。** 2048 对照实验结果是 null，`p=0.079`。这说明还没有证明普遍有效性，但也说明项目没有把机制存在包装成效果显著。这个诚实反而增强可信度。

### 6. 价值在哪里

它的价值不是“我做了一个万能自进化 Agent”。这句话太虚，也很容易被打穿。

真正价值有三层：

1. **工程价值**：让 Agent 不再每轮从零开始，减少重复试错，降低搜索浪费。
2. **可审计价值**：每条规则都能追到 `evidence_ids`。系统可以回答："为什么这个候选被屏蔽？是哪几次失败教会了这条规则？"
3. **方法论价值**：它把"失败"变成系统资产。以前失败只是日志，现在失败可以变成记忆、模式、规则、搜索约束和 pivot 信号。

### 7. 最稳的一句话总结

> 痕进不是要证明 Agent 已经无所不能。它做的是把 Agent 的试错过程变成可记忆、可审计、可复用的搜索引擎；现在能说的是机制存在、接入可行，但还不能说普遍有效。

这句话要背下来。它窄，但硬。

## 六、Q&A 应对手册

### Q1："这到底是 Agent 还是策略系统？"

> 回测器只是跑人写好的策略。这里不一样：Agent 自己决定挖哪个方向，也自己提出换池塘。那个判断——“不是因子死，是池塘错”——是它提出来的，我只是批准。普通回测器不会自己想到换市场。

### Q2："对照跑出 null，不就说明没用？"

> 不能这么理解。null 说明收益没有被统计检出，但机制本身确实在工作：完整组屏蔽了 452 个失败候选、触发 131 次 pivot，失忆组几乎全零。我区分的是“机制存在”和“效果显著”，不把前者包装成后者。

### Q3："这和 ChatGPT 记住对话有什么区别？"

> ChatGPT 的记忆更像把历史塞进 prompt 里碰运气。痕进做的是结构化：把失败归成模式，投影成规则，还带证据链。下次不是“想起来”，而是“查规则”。这是普通召回做不到的。

### Q4："动画里那个公式跳跃，是不是你预设好的？"

> 不是预设的。动画每一轮都是 `experiments/symreg_evolution.py` 真跑出来的，数据在 `experiments/_symreg_offset_s3.json`。公式是生成器从成功线索嫁接出来的，每一步的父代和操作都有记录，可以追到是哪一轮哪个公式“生”了它。

### Q5："这些公开领域都这么 toy，真实场景呢？"

> 第一个领域就是真实的：量化因子挖掘，私有数据，真实 case。换池塘那件事有 commit 历史，不是编的 demo。公开领域是为了可复现，因为私有数据评委验证不了。真实场景和公开可复现，两边都要。但我不会把“公开领域跑通”偷换成“已经证明泛化”。

### Q6（杀手锏问题）："你们的核心贡献到底是什么？"

> 三个字：**可审计**。别的系统说“Agent 学到了”，你问“哪几次失败教的”，它往往答不上来。我们能，因为每条规则带 `evidence_ids`。在一个满是黑箱学习曲线的赛道里，能回答“为什么”就是差异化。

### Q7："你的失败模式是谁定义的？人写的还是 Agent 学的？"

> 这要拆开说。motif 可以由人定义，规则可以从历史账本里投影出来；当前版本不是完全自动发现所有失败模式。我们的贡献不是说 Agent 凭空懂了领域，而是让“失败模式 → 规则 → 下一轮搜索”这条链变得结构化、可审计。

### Q8："这个引擎需要依赖大模型吗？跑起来贵不贵？"

> **引擎本身不依赖 LLM。** 它用 Python 标准库就能本地跑。它做的是结构化记录和检索：记失败、屏蔽死路、给家族打分、投影规则。这些都是确定性的数据结构操作，不需要调大模型。
>
> 你的项目里当然可以用 LLM 来生成候选，负责“造”。但痕进负责“筛和记”，这一层不依赖 LLM，所以成本和可复现性会稳很多。

## 七、高压追问清单（答辩时优先防守）

> **使用方式**：这些不是都要主动讲。只有被问到时，直接用下面的短答。别扩写，别越界。

### 1. "你到底证明了什么？"

> 我们现在能稳稳说的是：机制存在、规则可审计、引擎能跨领域接入。不能说的是：它已经在统计意义上普遍提升效果。这个边界我主动写在第 13 页。

### 2. "你们怎么判断这叫进化，而不是普通优化循环？"

> 我们不靠一句“结果变好了”来叫进化。我把它分三层：第一，历史有没有真的改变下一轮搜索；第二，系统有没有长出可追溯的新结构；第三，full 组是不是显著优于失忆组。现在前两层站得住，第三层在 2048 里是 null，所以不能 claim 普遍有效。

**展开版：**

- **机制层**：看历史是否被利用。指标包括 Memory 是否累积、失败候选是否被 block、是否投影出带 `evidence_ids` 的 Rule、下一轮排序是否改变、是否触发 pivot。2048 里完整组 block 452 个失败候选、pivot 触发 131 次，失忆组几乎为 0，说明机制层的进化行为存在。
- **行为层**：看系统是否长出新策略或新结构。符号回归里 `(x*x)` 经嫁接 `×2` 变成 `2*(x*x)`，R² 从 0.37 到 0.996；正则合成里 `.*` 收紧成日期规则。这说明它不是只会避错，也能从成功线索生成新候选。
- **效果层**：看使用历史的 full 组是否显著优于 amnesic/base 组。这个必须靠对照实验和统计检验。2048 的 `p=0.079` 没过显著性，所以效果层未被证明。

### 3. "p=0.079 的 null 对照，为什么还值得做？"

> 因为它把两件事分开了：机制有没有工作，收益有没有显著。完整组确实屏蔽了 452 个失败候选、触发 131 次 pivot；但性能提升没过显著性。这不是胜利叙事，是可信度叙事。

### 4. "和 Reflexion / ExpeL / Voyager / AgentEvolver 有什么区别？"

> 我不 claim 自进化机制是全新的。差异在工程形态：本地、确定性、可嵌入、不依赖 LLM，而且每条规则能追到 `evidence_ids`。别人说“学到了”，我们能回答“哪几次失败教的”。

### 5. "真实量化案例不能公开，怎么相信？"

> 私有案例只能说明它来自真实问题，不能当作公开可验证证据。所以我补了 2048、符号回归、正则合成、SQL、代码、RAG 这些公开材料。私有真实场景和公开可复现证据，各自承担不同职责。

### 6. "符号回归和正则是不是 toy？"

> 是 toy。这个我不回避。但它们不是用来证明生产价值的，而是让机制可见、可复现：从成功线索生成新候选，lineage 能追。生产价值要靠真实流水线评估，不能拿 toy 来冒充。

### 7. "为什么 2048 没显著，符号回归却看起来很好？"

> 因为它们回答的问题不同。2048 问的是“有没有显著提升”，结论是 null；符号回归问的是“能不能造出新候选”，结论是能看到轨迹。两个问题不能混着讲。

### 8. "这套东西什么时候会失效？"

> 四种情况不适合：失败不能客观定义、motif 不稳定、反馈太稀疏、任务是一次性的。痕进适合反复试错、同类错误会重复出现的搜索型任务，不适合所有 Agent。

### 9. "会不会误杀新方案，越来越保守？"

> 有这个风险，所以引擎里保留 novelty 和规则置信度，不是一刀切。更重要的是，每次 block / penalize 都可审计，真误杀了，也能追到是哪条规则造成的。

## 八、用户落地问题（产品化沟通）

### 1. "我接入它要改多少代码？"

> 通常不是改 core，而是写 adapter：候选怎么分类、特征怎么抽、失败 motif 怎么识别、结果怎么评估。简单领域几十到一百多行。难点不是代码量，是失败能不能定义清楚。

### 2. "它能不能直接提升我的 Agent 成功率？"

> 不能直接承诺。它先提供的是可记忆、可审计、可复用的搜索循环。成功率能不能提升，要在你的任务上做失忆组和完整组对照。

### 3. "我的业务失败能不能变成 motif？"

> 看三点：失败能不能客观判定，同类失败会不会重复出现，候选能不能抽出稳定特征。三点满足，motif 才有意义；否则只是贴标签。

### 4. "规则冲突怎么办？"

> 当前规则是结构化对象，有 action、confidence、score_adjustment 和 evidence_ids。冲突可以按置信度、作用域和动作优先级处理。关键是它不是黑箱冲突，每条规则都能解释来源。

### 5. "能不能接 LangChain / AutoGen / 自研 Agent？"

> 可以，但痕进不替代这些框架。它插在候选生成和执行之间：上游给候选，下游执行评估，痕进负责排序、屏蔽、记录和规则投影。

### 6. "它到底是产品、库，还是研究原型？"

> 现在更准确地说，是开源研究原型加一个可嵌入引擎。它已经证明机制和接入边界，但还不是开箱即用的平台产品。

## 九、三句保命回答

1. **我们现在证明的是机制存在、可审计、可跨领域接入；还没证明普遍有效性。**
2. **这不是替代 Agent 的平台，而是嵌入搜索循环里的记忆和规则层。**
3. **难点不在 core，而在把你的领域失败翻译成可靠 motif；这一步我们不装傻。**
