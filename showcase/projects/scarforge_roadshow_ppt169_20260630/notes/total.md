# 01_封面

大家好，我是乔木计划，今天给大家介绍的项目叫"痕进"，英文名 Scarforge。一句话说清楚它是什么：一个给 Agent 装上的进化层，它的作用是把失败变成下一次行动的约束。听起来有点反直觉，大家可能觉得失败就该被清掉，但我们今天的核心观点恰恰相反——失败是 Agent 最值钱的资产，只是过去没人好好用它。

---

# 02_承诺

先说我们承诺的一件事。痕进这个名字，"失败留痕，凭痕而进"。注意它不是让 Agent 背更多的上下文，现在很多人做大上下文窗口就能解决问题，我们不是这条路。我们让它学会长记性：错在哪里、为什么错、下次该避开哪条路，都留下能追溯的痕迹。这个"可追溯"三个字很重要，后面会反复出现。

---

# 03_问题

问题出在哪呢？今天的 Agent 太容易反复犯同一个错。客服也好、写代码也好、做搜索也好，都一样——一次失败刚修好，下一次换个壳又回来了。我们认为根本原因不只是记忆不够，而是失败根本没进入决策系统。具体有三种典型做法：只记成功，把错误路径当垃圾删掉；只调参数，prompt 改来改去，但该换的任务结构没换；只追最优，大家都挤向看起来最好的方向，没被试过的新路线就被放弃了。

---

# 04_五机制

那我们的论点是什么？不是去修 prompt，是让失败变成系统资产。一次失败如果只停在对话窗口里，它就是噪音；只有当它被写进账本、归成模式、投影成规则、去影响下一轮搜索，才开始产生复利。所以我们设计了五个机制：第一记住失败，把失败持久化，别把配额浪费在已知死路上；第二评分搜索，评估"怎么找答案"的过程，而不只看最后答案对不对；第三学习模式，从"这次错了"升级到"这类情况都会错"；第四换结构，方法反复失败时就换路径，别在旧路上微调；第五避拥挤，奖励新颖探索，防止集体挤进局部最优。这五个机制是一个闭环。

---

# 05_沙盘

光说机制太空，我们拿一个真实沙盘来验证。量化交易是我们的第一个沙盘，因为它的反馈明确、失败可追溯。这个案例不是想证明我们会交易，而是证明 Agent 学会了换搜索结构。看这两组对比：左边是大盘池塘，Sharpe 是二点一五，但自相关性高达零点七八，结果失败；右边是中小盘池塘，同样 Sharpe 二点一五，自相关性降到零点六八，结果通过。关键转折在哪？答案没变，是环境变了——由 Agent 提出建议，人来批准，这是人机协同的一次真实换结构。

---

# 06_可审计性

有人可能会问，这不是黑箱学习吗？我们说不是，因为每一条规则都能追到证据。看这个例子：对 cashflow 这个因子施加零点六五的惩罚，置信度零点七五，证据来自三次失败。真正的差异不在于"Agent 好像学到了东西"，而在于它能回答你：到底是哪几次具体的失败，教会了它这条惩罚。能回答这个问题，它才不是黑箱。

---

# 07_架构

再看产品形态。痕进是一个可嵌入的进化层，不是又一个 Agent 平台。你的 Agent 继续负责干活，痕进只负责让它从失败中更新搜索方式。架构很简单，两层：上层是 core，通用进化循环，记录失败、学习模式、更新规则、奖励新颖探索；下层是 adapter，领域知识的薄层，解释什么叫失败、什么叫好结果、什么动作能执行。换领域的时候，只换 adapter，进化循环永远留在 core。

---

# 08_应用场景

这个机制能用在哪些地方？我们的判断是：凡是会试错的 Agent，都需要一套失败进化机制。量化交易只是第一个沙盘。代码 Agent，编译错误和测试失败被归成模式，下次调整的是代码结构而不是重复打补丁；自动化 Agent，记住哪类输入容易触发问题；检索 Agent，搜索结果差的时候进化查询路径，而不是只换几个关键词；教育 Agent，学生反复出错时识别错误类型，调整讲解顺序；模拟 Agent，在仿真环境里积累试错痕迹，更新下一轮策略。本质上，痕进服务的是所有"可重复试错"的工作流。

---

# 09_诚实边界

讲到这里，我们必须把边界说清楚，因为真正的可信感来自边界清晰。我们 claim 三件事：五个机制都已经实现、我们观察到一次真实的换结构事件、部分学习确实来自账本投影而不是手写规则。同时也有三件事我们明确不 claim：没有有效性的对照证明、没有量化收益曲线、也还没有跨领域规模化的验证结论。这不是软弱，这是一个可信产品该有的诚实。我们讲的是一个可追溯的进化事件，不把它包装成万能神话。

---

# 10_收尾

最后用一句话收尾。大多数 Agent 把失败当垃圾，痕进把失败变成下一次行动的路标。真正的智能，不是第一次就做对，而是第一次做错之后，第二次少走同一条错路。这就是我们正在做的事，谢谢大家。
