# 痕进引擎 · 项目接入指南

> **这份文档回答：怎么把痕进引擎用到一个具体项目里。**
> 从"我的项目适不适合"到"代码怎么写"到"多轮怎么跑"，全程有可运行的代码示例。

---

## 一、先判断：我的项目适不适合？

### 三个必要条件（必须同时满足）

| 条件 | 含义 | 反例（不适合） |
|---|---|---|
| **反复试错** | Agent 不是做一次，而是多轮搜索 | 写一封邮件（一次性） |
| **失败可定义** | 有客观的"行/不行"标准 | 文章写得好不好（主观） |
| **同类错误重复** | 同一类失败会再次出现 | 每次都是全新问题 |

### 适合的真实场景

| 场景 | 试错循环 | 失败定义 | 痕进的价值 | 成熟度 |
|---|---|---|---|---|
| **量化因子挖掘** | 提因子→回测→看指标 | Sharpe 低、拥挤 | 记住"这类信号在这个市场失效"，换方向 | ✅ 已验证 |
| **符号回归** | 猜公式→拟合→看 R² | R² 低 | 记住"三角函数族拟合不了多项式" | ✅ 已验证 |
| **正则/规则合成** | 猜规则→分类→看 F1 | 假阳性/假阴性 | 记住"过宽的规则会误伤"，收紧 | ✅ 已验证 |
| **代码生成** | 写代码→编译测试→修 | 编译错误、测试失败 | 记住"这类 API 调用易触发异常" | 🔜 Roadmap |
| **SQL 查询优化** | 写查询→执行→看性能 | 超时、全表扫描 | 记住"这类 JOIN 在大表上必慢" | 🔜 Roadmap |
| **RAG 检索** | 检索→生成→评估 | 幻觉、答非所问 | 记住"这类查询召回差"，进化策略 | 🔜 Roadmap |

## 二、引擎在项目里的位置

```
你的项目
  │
  ├── 你的 Agent / 生成器（负责"造"候选）
  │     └── 可以是 LLM、算法、随机生成器——引擎不管
  │
  ├── 痕进 engine（负责"筛和记"）
  │     └── core/  ← 通用引擎，一行不改，纯 stdlib，不依赖 LLM
  │
  └── 你的评估器（负责"判"结果好坏）
        └── 编译器、测试框架、回测引擎——你项目本来就有的
```

**引擎是中间件**，夹在"生成器"和"评估器"之间：
- 生成器产出候选 → 引擎根据记忆**屏蔽已知死路**、**排序** → 评估器执行 → 引擎**记录结果** → 下一轮生成器从引擎的反馈里获益

## 三、接入只需 6 步（附可运行代码）

### 第 1 步：安装

```bash
# 克隆仓库，把 core/ 和 adapters/mock/ 拿走
git clone https://github.com/Joe-rq/evolving-agents.git
# core/ 是引擎核心，你的项目只需要这一个目录
```

引擎零外部依赖——只用 Python 标准库（`random`, `math`, `dataclasses`, `collections`），不需要 `pip install` 任何东西。

### 第 2 步：定义你的候选类型

一个候选就是你领域里的一个"方案"——一个公式、一段代码、一条 SQL、一个因子表达式。它需要一个 `body` 字符串（用于去重）：

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class CodeVariant:
    code: str           # 代码内容
    family: str         # 策略家族："recursive" / "iterative" / "functional"
    features: list      # 特征：["api:requests", "flow:for", ...]
    motif: str          # 预判失败模式："null_unsafe" 或 ""
    @property
    def body(self):     # 规范化字符串，引擎用它去重
        return self.code
```

### 第 3 步：写 adapter（5 个方法，约 50-150 行）

adapter 是你的领域知识薄层——告诉引擎怎么理解你的候选：

```python
from core.protocols import DeadEndContext

class CodeGenAdapter:
    """把你的领域知识映射成引擎能理解的 5 个方法。"""

    def classify(self, candidate):
        """Factor 2 — 把候选分成'家族'，引擎按家族打分。
        返回一个字符串，如 'recursive' / 'iterative' / 'functional'。"""
        return candidate.family

    def extract(self, candidate):
        """Factor 1/5 — 提取特征，用于去重 + 新颖性打分。
        返回一个字符串列表，如 ['api:requests', 'flow:for']。"""
        return candidate.features

    def motif_of(self, candidate):
        """Factor 3 — 预判这个候选的'失败模式'（一类容易一起失败的候选）。
        返回一个字符串如 'null_unsafe'，或空字符串 ''。"""
        return candidate.motif

    def dead_end_reason(self, candidate, context: DeadEndContext):
        """Factor 1 — 这个候选是不是已知死路？
        context.tested_bodies 是之前测过的 body 集合。
        返回非空字符串='屏蔽原因'，空字符串='可以试'。"""
        if candidate.body in context.tested_bodies:
            return "already_tested"
        return ""

    def is_success(self, result):
        """结果是不是成功？引擎根据这个决定记什么为'赢'。"""
        return result.get("success", False)

    def is_weak(self, result):
        """结果是不是'接近但不够'？这是介于成功和失败之间的信号。"""
        return result.get("weak", False)
```

### 第 4 步：定义失败模式规则

告诉引擎"哪种 motif 积累够多次失败就惩罚/屏蔽"：

```python
from core.rules import RuleSpec

YOUR_RULES = [
    # motif,     evidence_kind,    min_evidence, action,     adjustment, reason
    RuleSpec("null_unsafe", "compile_error", 3, "penalize", -0.5,
             "无 null 检查的访问反复触发编译错误"),
    RuleSpec("sql_injection", "security_fail", 1, "block", -1.0,
             "拼接 SQL 字符串有注入风险，直接屏蔽"),
]
```

### 第 5 步：写你的 execute（评判函数）

这是你项目本来就有的——编译代码、跑测试、执行查询：

```python
def my_execute(candidate):
    """执行一个候选，返回结果字典。
    引擎只关心 success / weak / evidence_kinds 这三个字段。"""
    try:
        result = compile_and_test(candidate.code)
        return {
            "success": result.tests_passed and not result.compile_errors,
            "weak": result.compiled and not result.tests_passed,
            "evidence_kinds": result.failure_tags,  # 如 {"compile_error", "null_deref"}
        }
    except Exception as e:
        return {"success": False, "weak": False, "evidence_kinds": {"exception"}}
```

### 第 6 步：组装引擎，跑多轮循环

```python
from core.engine import Engine
from core.memory import Memory

# 组装（一次性）
adapter = CodeGenAdapter()
memory = Memory()
engine = Engine(adapter, memory, YOUR_RULES)

# 多轮循环（每轮是一次搜索迭代）
for round_idx in range(N_ROUNDS):
    # 1. 你的生成器产出候选（可以是 LLM、算法、随机）
    candidates = your_generator.generate(round_idx)

    # 2. 引擎筛选+执行+记录
    result = engine.run(
        candidates=candidates,
        universe="your_project_name",     # 环境标识（不同环境的记忆隔离）
        execute=my_execute,               # 你的评判函数
        max_execute=5,                    # 每轮最多试几个
    )

    # 3. 引擎自动做了这些（你不用管）：
    #    - 屏蔽已知死路（dead_end_reason）
    #    - 按家族分数 + 新颖性排序
    #    - 执行 top N
    #    - 记录每个候选的成功/失败/证据
    #    - 投影规则（积累够失败就惩罚整个 motif 类）
    #    - 检查是否该换结构（pivot）

    # 4. 你可以根据结果调整下一轮的生成
    best = max(result["executed"], key=lambda x: x[1].get("score", 0))
    your_generator.feedback(best, result["pivot_recommendation"])
```

## 四、多轮循环里发生了什么（进化的核心）

| 轮次 | Memory 状态 | 引擎行为 | 效果 |
|---|---|---|---|
| 第 1 轮 | 空 | 全部执行，记录所有结果 | 知道哪些方向失败 |
| 第 2 轮 | 记住了第 1 轮 | 屏蔽已测过的 body | 不重复试已知答案 |
| 第 3 轮 | motif 失败累积够了 | 投影出规则，惩罚整个失败类 | 整类跳过，省大量试错 |
| 第 N 轮 | 多条规则 + 家族分数 | 家族分数指导排序 | 往高成功率方向倾斜 |
| 某轮 | 2+ 家族全失败 | 触发 pivot 建议 | "该换结构了，不是调参数" |

**关键：Memory 跨轮累积——这就是"从失败里学习"。** 没有痕进，每轮都从零开始；有痕进，每轮都站在前几轮的肩膀上。

## 五、不需要 LLM——但可以配合 LLM 用

引擎本身**零 LLM 依赖**（纯 stdlib，本地跑，免费）。但你的项目可以这样配合 LLM：

```
LLM 生成 5 个代码方案
       ↓
   痕进引擎：屏蔽已知会失败的写法，只保留没试过的
       ↓
   编译测试：执行 + 评判
       ↓
   痕进引擎：记录"这类写法失败了"
       ↓
   下一轮：LLM 生成的方案里，引擎自动跳过失败过的模式
```

**LLM 负责"造"（生成候选），引擎负责"筛和记"（屏蔽死路 + 记录模式）。** 引擎不调 LLM，但帮 LLM 避免重复犯错。

## 七、常见问题

**Q：引擎怎么知道"哪个方向有希望"？**
A：通过 `MetaLearner.score_families()`——它统计每个家族（你用 `classify` 定义的）的历史成功率，高成功率的家族排序靠前。这是 Factor 2（给搜索打分）。

**Q：引擎怎么知道"该换结构了"？**
A：通过 `Evolver.should_pivot()`——当 2 个以上家族全部失败（strike），或者某个 motif 重复失败超过阈值，它会触发 pivot 建议。这是 Factor 4（换结构）。

**Q：Memory 会无限增长吗？**
A：当前是 append-only。生产环境可以加 TTL 或滑动窗口，但研究阶段全量保留是为了可审计性——每条规则都要追到 evidence_ids。

**Q：不同项目的记忆会互相污染吗？**
A：不会。Memory 是 universe-scoped 的——`engine.run(candidates, universe="project_A")` 的记忆和 `universe="project_B"` 完全隔离。这就是换池塘案例里 TOP3000 的失败不污染 TOP1000 的机制。

**Q：我只想用"记住失败+屏蔽"，不想搞家族打分，行吗？**
A：行。adapter 的 `classify`/`motif_of` 可以返回简单值，规则列表可以只放一条 block 规则。引擎会优雅降级——只做 Factor 1（记住+屏蔽），不做 Factor 2/3。
