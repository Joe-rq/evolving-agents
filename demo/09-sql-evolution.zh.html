一条进化链：每一步修掉一个反模式

从最坏的查询（4 个反模式，cost=24）开始，生成器每轮嫁接一个修复操作。代价从 24 一路降到 1.0（最优）。
24.0
SELECT * FROM orders JOIN customers
反模式：SELECT * / 笛卡尔积 JOIN / 无 ON / 无 LIMIT
↓ graft: +ON — 给 JOIN 加上连接条件，消除笛卡尔积
8.0
SELECT * FROM orders JOIN customers ON orders.id = customers.id
反模式：SELECT * / 无 LIMIT（笛卡尔积已修）
↓ graft: *→id — 把 SELECT * 收紧为 SELECT id
3.0
SELECT id FROM orders JOIN customers ON orders.id = customers.id
反模式：无 LIMIT
↓ graft: +LIMIT — 加 LIMIT 限制结果集
1.0
SELECT id FROM orders JOIN customers ON orders.id = customers.id LIMIT 1
✅ 无反模式 — 最优查询
每一步修掉一个反模式，代价从 24 降到 1。 graft 操作（+ON / *→id / +LIMIT）都是 SQL 优化里的真实操作——这不是玩具，是生产 DBA 每天做的事。引擎把这些操作从"人工经验"变成了"从失败里自动学"。

诚实边界

✅ 证明了⚠️ 局限
引擎在第五个领域（SQL 优化）跑通；进化链清晰：24→8→3→1，每步修一个反模式。这是生产相关场景，不是玩具。 代价模型是简化版（规则估算，非真实执行计划）。真实 DB 的代价取决于数据分布、索引、统计信息——这里用静态规则替代。
graft 操作（+ON / *→id / +LIMIT）是真实的 SQL 优化手段。 FULL vs BASE 分化不如符号回归稳定——SQL 简单查询太容易达到 cost=1.0（天花板效应）。展示用单组进化链而非分化曲线。
定位：SQL 证明了引擎在生产相关任务上的可接入性。它不是"最强进化案例"（那是符号回归），而是"这套机制能优化你每天写的 SQL"的直接证据——从 Roadmap 变成了已验证。
